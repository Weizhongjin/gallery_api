# 图片导入运行规范（Operating Model）

## 目标

为保障稳定性与可维护性，Gallery API 采用分层导入模式：

1. 大批量导入由后台 CLI 负责
2. 网页/软件仅承担小批量上传
3. 网页/软件主要承担检索、浏览、审核与发布

---

## 角色分工

### A. 后台 CLI（主通道）

适用场景：
- 历史数据初始化
- 季度批量上新（成百上千张）
- 回填 AI（classify / embedding）

能力要求：
- 并发上传
- 断点续跑
- 失败重试
- 可审计日志
- 批次统计（成功/失败/跳过）

结论：
- 批量导入 **必须优先走 CLI**

### B. 网页/软件（辅通道）

适用场景：
- 日常补图
- 小批量修正（几十张级别）
- 运营手工上传

结论：
- 网页/软件只做轻量上传，不承担大规模搬运

### C. 网页/软件（核心使用面）

主要功能：
- 条件筛选与检索（属性检索 + 向量检索）
- 分组浏览与详情查看
- 标签审核与修正
- Lookbook 编排与发布

---

## 三档图片策略

每个资产应维护三种图片：

1. `original`：原始图
2. `display`：中间尺寸图（用于 VLM / Embedding / 详情展示）
3. `thumb`：缩略图（用于列表与瀑布流）

当前规则：
- 若 `original > 3MB`，必须生成独立 `display/thumb`
- Embedding 固定使用 `display_uri`
- 若 `original > 3MB` 且 `display_uri == original_uri`，处理应报错（阻断不合规数据）

---

## 推荐流程

1. CLI 导入原图并生成 `display/thumb`
2. 写入 `asset` 与分组元数据
3. 触发 AI 流程（`classify` / `embed`）
4. 网页查看任务进度、失败明细与结果
5. 对 `taxonomy_candidate` 进行人工审核与提升

---

## 接口定位建议

保留并强化以下边界：

1. API 对外提供“任务触发 + 状态查询”
2. 大规模文件搬运放在 CLI / worker，不放在浏览器长连接中
3. UI 以“管理与检索”为主，不以“大批量传输”为主

---

## 运维建议

1. 批量导入任务记录批次 ID、执行人、时间窗
2. 每批次导入后自动输出质量报告（尺寸合规、特征完成率、失败重试结果）
3. 将“>3MB 但 display 未分离”作为巡检项，按天告警
